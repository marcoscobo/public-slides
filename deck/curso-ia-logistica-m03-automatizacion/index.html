n8n

La plataforma de automatización que pone el poder del código en manos de cualquiera.

QUÉ ES N8N

Automatización visual, código opcional

n8n es una plataforma de automatización de flujos de trabajo open-source y self-hosteable. Permite conectar aplicaciones, APIs y bases de datos mediante un editor visual de nodos, sin necesidad de escribir código.

Cada flujo es un conjunto de nodos enlazados: un nodo dispara el flujo (trigger), el resto lo procesan. Los datos viajan como JSON entre nodos, y cada uno puede filtrar, transformar o enrutar el flujo según lógica condicional.

ARQUITECTURA

Los nodos que mueven tu logística

  • 01

    Trigger nodes

    Webhook, Schedule, Email, Form. Arrancan el flujo cuando ocurre un evento externo o en un horario definido.

  • 02

    Action nodes

    HTTP Request, Send Email, Google Sheets, Slack, WhatsApp. Ejecutan una acción sobre un sistema externo.

  • 03

    Logic nodes

    IF, Switch, Merge, Loop, Wait. Controlan el flujo según condiciones, ramifican y agregan datos.

  • 04

    Transform nodes

    Set, Function, Code, JSON Parse. Modifican, enriquecen o filtran los datos en tránsito.

CAPÍTULO 02

Correos automáticos

Alertas de stock, confirmaciones de pedido y notificaciones de entrega que se envían solos.

CASOS DE USO · CORREO

Tres alertas que nunca deben perderse

  • 01

    Alerta de stock bajo

    n8n consulta el WMS cada hora. Si un SKU cae por debajo del mínimo, envía email al comprador y crea una orden de reposición en el ERP automáticamente.

  • 02

    Confirmación de pedido

    Al registrar un nuevo pedido en el ERP, el flujo genera un email personalizado con número de pedido, artículos, fecha estimada de entrega y enlace de seguimiento.

  • 03

    Notificación de entrega

    Webhook del TMS dispara el flujo al cambiar el estado del envío. El cliente recibe un SMS y un email con el código de seguimiento en tiempo real.

EJEMPLO · ALERTA DE STOCK

Flujo de alerta en 6 nodos

// Nodo 1: Schedule Trigger (cada hora)
cronExpression: "0 * * * *"

// Nodo 2: HTTP Request → WMS API
GET /api/stock?threshold=minStock

// Nodo 3: IF → stock < minimum
{{ $json.quantity }} < {{ $json.minStock }}

// Nodo 4: Send Email → comprador
subject: "Stock bajo: {{ $json.sku }}"

// Nodo 5: HTTP Request → ERP (crear orden)
POST /api/purchase-orders

Sin código real. Cada paso es un nodo visual; el pseudocódigo muestra la lógica del flujo.

Condicional. El nodo IF evita envíos innecesarios cuando el stock está bien.

Doble acción. Un solo flujo notifica Y actúa en el ERP.

CAPÍTULO 03

Procesamiento de pedidos

Del canal de venta al almacén, sin que nadie tenga que copiar y pegar datos a mano.

FLUJO DE PEDIDO AUTOMÁTICO

De la recepción a la confirmación

  1. 01

    Recepción

    Webhook del canal de venta (web, marketplace, EDI) dispara el flujo con los datos del pedido en JSON.

  2. 02

    Validación

    n8n comprueba stock disponible en el WMS, verifica datos del cliente y detecta duplicados antes de proceder.

  3. 03

    Registro en ERP

    El pedido se crea automáticamente en el ERP con todos los campos mapeados. Sin intervención del equipo de back-office.

  4. 04

    Confirmación

    Email personalizado al cliente, notificación interna en Slack al equipo de almacén y actualización del CRM. Todo en segundos.

ARQUITECTURA DE INTEGRACIÓN

n8n como capa de orquestación

En lugar de integraciones punto a punto (n sistemas = n² conexiones), n8n actúa como hub central. Cada sistema habla con n8n, y n8n decide qué datos enviar a quién y cuándo.

Los mecanismos de conexión son: API REST (la mayoría de ERP/WMS modernos), Webhooks (notificaciones push en tiempo real) y bases de datos directas (PostgreSQL, MySQL) cuando no hay API disponible.

TÉCNICAS DE INTEGRACIÓN

Cuatro formas de conectar sistemas

  • 01

    API REST + autenticación OAuth

    SAP, Sage, Odoo, Holded. n8n gestiona los tokens automáticamente y reintenta en caso de error de red.

  • 02

    Webhooks bidireccionales

    El TMS avisa a n8n cuando cambia el estado de un envío. n8n avisa al ERP para actualizar el pedido. Latencia menor de 5 segundos.

  • 03

    Consulta directa a base de datos

    Para sistemas legacy sin API, el nodo Postgres/MySQL de n8n consulta directamente las tablas del WMS y transforma los datos al vuelo.

  • 04

    Ficheros EDI / CSV automatizados

    n8n monitoriza una carpeta SFTP o un bucket S3, procesa los ficheros EDI al llegar y los inyecta en el ERP sin intervención manual.

CAPÍTULO 05

Agentes autónomos

El siguiente nivel: sistemas que no solo automatizan, sino que razonan y toman decisiones.

DEFINICIÓN · AGENTE LOGÍSTICO

Más allá de si-entonces

Un agente logístico autónomo es un sistema de IA que percibe el estado de la operación, razona sobre qué acción tomar y la ejecuta, repitiendo el ciclo sin intervención humana.

En n8n, se construye combinando el nodo AI Agent (con un LLM como cerebro) con herramientas: consultar el WMS, crear órdenes en el ERP, contactar proveedores o escalar incidencias al equipo según la gravedad.

CASOS DE USO AVANZADOS

Lo que un agente puede gestionar solo

  • 01

    Gestión de incidencias de entrega

    El agente detecta un paquete retenido, consulta el TMS, evalúa el SLA, contacta al cliente y reasigna el envío a un transportista alternativo si es necesario.

  • 02

    Reposición predictiva de stock

    Analiza histórico de ventas, estacionalidad y lead times de proveedores para calcular el punto de reorden óptimo y lanzar la orden de compra de forma autónoma.

  • 03

    Optimización dinámica de rutas

    Cada mañana, el agente consulta pedidos del día, agrupa por zona, llama a la API del TMS y genera las rutas optimizadas para el equipo de reparto.

  • 04

    Atención a clientes sobre pedidos

    Chatbot con acceso a ERP y TMS que responde en tiempo real: estado del envío, cambio de dirección, cancelaciones y generación de devoluciones.

De automatización a agente IA con n8n

      │
      ▼
  Parse datos (nodo Set)

      ▼
  IF stock > 0  ──→  Registrar ERP  ──→  Enviar email

── AGENTE IA (bucle de razonamiento) ─────────────────
  Trigger (webhook pedido)
      │
      ▼
  Parse datos (nodo Set)
      │
      ▼
  Nodo AI Agent ◄──────────────────────────────────┐
  "Decide: aprobar, escalar o rechazar           │
   según reglas de negocio y contexto"           │
      │   ◄── herramientas disponibles:              │
      │        · consultar_stock(sku)                 │
      │        · verificar_credito(cliente_id)        │
      │        · obtener_historial(pedido_id)         │
      │                                               │
      ├──→ APROBAR  → ERP + email confirmación          │
      ├──→ ESCALAR  → Slack equipo + ticket Jira        │
      └──→ RECHAZAR → email motivo + log auditoría       │
               ↑                                      │
               └──── necesita más info ───────────────┘

Automatización clásica. Lógica fija: si A entonces B. Rápida pero rígida — no gestiona excepciones ni contexto.

Agente IA. El nodo AI Agent (LLM) razona sobre los datos del pedido, el historial del cliente y el stock antes de decidir. Puede pedir más información antes de actuar.

Bucle de razonamiento. El agente itera hasta tener suficiente contexto; la automatización clásica ejecuta una sola vez.

Ejemplo concreto. Pedido llega → IA detecta que el cliente tiene una deuda pendiente → escala al equipo de crédito en vez de aprobar automáticamente.
