CHAPTER 01

OCR Inteligente

De los píxeles a los datos: la evolución del reconocimiento óptico de caracteres y el salto al procesamiento inteligente de documentos.

EVOLUCIÓN TECNOLÓGICA

Del OCR clásico al IDP

  1. 1990s

    OCR clásico (reglas)

    Reconocimiento por patrones fijos. Requería documentos perfectos, plantillas rígidas y alta supervisión manual.

  2. 2010s

    OCR con ML

    Redes neuronales convolucionales mejoran la tolerancia al ruido. Primeros modelos de reconocimiento de campos semánticos.

  3. 2018+

    Document AI / IDP

    Transformers multimodales (texto + layout + imagen). Comprensión semántica, extracción contextual, sin plantillas.

  4. HOY

    LLMs documentales

    Modelos como GPT-4V o Claude para documentos complejos. Razonamiento sobre contenido, no solo extracción.

HERRAMIENTAS DEL MERCADO

Modelos OCR actuales

  • 01

    Tesseract (Open Source)

    Motor gratuito mantenido por Google. Bueno para documentos limpios, limitado en layouts complejos. Ideal para pruebas y volúmenes bajos.

  • 02

    AWS Textract

    Extrae texto, tablas y formularios de PDFs e imágenes. API managed, pay-per-use. Integración nativa con el ecosistema AWS.

  • 03

    Azure Form Recognizer

    Modelos preentrenados para facturas, recibos y documentos de identidad. Permite entrenamiento custom con pocos ejemplos.

  • 04

    Google Document AI

    Suite completa: OCR, clasificación, extracción y enriquecimiento. Procesadores especializados por tipo de documento.

CHAPTER 02

Facturas

Automatizar la validación, extracción y reconciliación de facturas con órdenes de compra: el caso de uso más maduro del IDP en logística.

ANATOMÍA DE LA FACTURA

Campos clave para la extracción

Una factura logística contiene datos estructurados en zonas predecibles. El IDP aprende la semántica de cada campo, no su posición exacta en la página.

Campos críticos: número de factura, fecha de emisión, emisor/receptor (CIF, razón social), líneas de detalle (cantidad, descripción, precio unitario, IVA), totales, número de PO referenciado y condiciones de pago.

VALIDACIÓN AUTOMÁTICA

Extracción vs reconciliación

EXTRACCIÓN

Datos del documento

  • OCR + NER sobre campos semánticos
  • Confianza por campo (0–100%)
  • Detección de anomalías tipográficas
  • Normalización de formatos (fechas, monedas)
RECONCILIACIÓN CON PO

Cruce con ERP/WMS

  • Matching por número de PO referenciado
  • Tolerancia configurable (± 2% en importes)
  • Alerta automática en discrepancias
  • Flujo de aprobación para excepciones

CHAPTER 03

Albaranes

Digitalización en tiempo real, validación contra pedidos y trazabilidad total en el almacén.

PIPELINE DE DIGITALIZACIÓN

Albarán digital: del papel al WMS

  • 01

    Captura en recepción

    Foto con móvil o escáner en el muelle. La imagen se envía al pipeline IDP en tiempo real (latencia <3 seg).

  • 02

    Extracción y validación

    El modelo extrae líneas de pedido, referencias, cantidades y fechas. Se cruzan contra el pedido abierto en el ERP.

  • 03

    Integración WMS

    Si la validación es correcta, el WMS crea automáticamente el movimiento de stock. Sin teclear una línea.

  • 04

    Trazabilidad completa

    Cada albarán digitalizado queda enlazado a la expedición, el pedido y la factura. Auditoría completa en un clic.

CHAPTER 04

CMR

La Carta de Porte internacional: campos obligatorios, extracción automática y beneficios en el transporte de mercancías por carretera.

CARTA DE PORTE INTERNACIONAL

CMR: campos obligatorios y extracción automática

El CMR (Convenio Relativo al Contrato de Transporte Internacional de Mercancías por Carretera) define 24 casillas normativas. El IDP extrae todas sin plantillas fijas.

Campos críticos: remitente y destinatario, lugar y fecha de carga, descripción de la mercancía, peso bruto, instrucciones para despacho de aduanas, número de matrícula del vehículo, firma y sello del transportista.

Beneficios: reducción de disputas por datos incorrectos, integración automática con TMS, trazabilidad en tiempo real y cumplimiento normativo automatizado.

Clasificación

Modelos que identifican el tipo de documento automáticamente antes de procesarlo: el guardabarros inteligente del pipeline documental.

CLASIFICACIÓN DE DOCUMENTOS

Modelos y casos logísticos

CÓMO SE ENTRENA

Arquitectura del modelo

  • Fine-tuning de LayoutLM o DistilBERT
  • Features visuales (layout) + textuales
  • Pocas muestras: 50–200 ejemplos por clase
  • Reentrenamiento continuo con feedback
CASOS EN LOGÍSTICA

Clases habituales

  • Factura de proveedor nacional / internacional
  • Albarán de entrega / devolución
  • CMR, AWB (áereo), BL (marítimo)
  • Certificados de origen, packing lists

CHAPTER 06

Extracción

NER, modelos transformer y pipelines de procesamiento: la maquinaria que convierte texto no estructurado en datos accionables.

TECNOLOGÍAS DE EXTRACCIÓN

NER y Transformers documentales

  • 01

    NER (Named Entity Recognition)

    Identifica y clasifica entidades: fechas, importes, empresas, referencias, direcciones. Modelos spaCy o Flair sobre texto extraído por OCR.

  • 02

    LayoutLM / LayoutLMv3

    Transformers que combinan texto, posición en la página e imagen. Estado del arte para extracción en documentos con layouts variables.

  • 03

    Donut (Document Understanding Transformer)

    End-to-end: imagen de entrada, JSON de salida. Sin OCR separado. Excelente para formularios y documentos semiestructurados.

  • 04

    LLMs como extractor general

    GPT-4V o Claude con prompts estructurados extraen cualquier campo con razonamiento contextual. Flexible pero con coste y latencia mayores.

IMPLEMENTACIÓN

Pipeline IDP en Python

import boto3
from transformers import pipeline

textract = boto3.client('textract', region_name='eu-west-1')
ner = pipeline('ner', model='dslim/bert-base-NER')

def process_document(image_bytes):
    # 1. OCR con AWS Textract
    ocr = textract.detect_document_text(
        Document={'Bytes': image_bytes}
    )
    text = ' '.join(
        b['Text'] for b in ocr['Blocks']
        if b['BlockType'] == 'LINE'
    )
    # 2. Extracción de entidades con NER
    entities = ner(text)
    return entities

Paso 1 — OCR. Textract devuelve bloques de texto con posición y confianza.

Paso 2 — NER. El modelo BERT fine-tuneado identifica empresas, fechas e importes.

Siguiente paso. Post-procesado para mapear entidades a campos del esquema de destino (ERP/WMS).

PASOS DEL PIPELINE
  • Paso 1 — OCR (Tesseract / AWS Textract): extrae texto plano con coordenadas de cada token
  • Paso 2 — LLM (Claude): recibe el texto OCR y devuelve JSON estructurado con campos semánticos

  • Output: JSON validado listo para importar al ERP vía API

El prompt enviado a Claude y su respuesta JSON

# Prompt enviado a Claude (con few-shot)
prompt = """Eres un extractor de datos de facturas logísticas.
Dado el siguiente texto extraído por OCR de una factura,
devuelve ÚNICAMENTE un JSON con los campos indicados.
Si un campo no existe, usa null.

EJEMPLO:
Texto OCR: "FACTURA 2024-001 ACME SL B-12345 15/01/2024
            Transporte Madrid 500.00 EUR IVA 21%"
JSON: {"factura_num": "2024-001", "proveedor": "ACME SL",
       "cif": "B-12345", "fecha": "2024-01-15",
       "concepto": "Transporte Madrid",
       "base_imponible": 500.00, "iva_pct": 21,
       "total": 605.00}

TEXTO OCR A PROCESAR:
{texto_ocr}

JSON:"""

# Llamada a la API
import anthropic
client = anthropic.Anthropic()
response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{"role": "user",
               "content": prompt.format(texto_ocr=texto)}]
)

Few-shot. El ejemplo incluido en el prompt actúa como referencia de formato. Claude generaliza a cualquier factura sin entrenamiento adicional.

JSON garantizado. Pedir explícitamente "devuelve ÚNICAMENTE un JSON" reduce el riesgo de texto extra que rompa el parser.
