CAPÍTULO 01

Vehicle Routing Problem

El problema combinatorio que define la logística de última milla — y cómo la IA lo ataca.

FUNDAMENTOS

¿Qué es el VRP?

El Vehicle Routing Problem consiste en encontrar el conjunto óptimo de rutas para una flota de vehículos que debe servir a un grupo de clientes dispersos geográficamente, minimizando el coste total.

Es una generalización del Travelling Salesman Problem (TSP): en vez de un único viajero, hay múltiples vehículos que parten y regresan a uno o varios depósitos. La complejidad computacional es NP-hard — una red de 100 clientes tiene más de 10¹⁵⁷ rutas posibles.

En logística real, el VRP se traduce directamente en decisiones diarias: ¿cuántos camiones salen hoy?, ¿en qué orden visitan los puntos?, ¿qué carga lleva cada uno? Un ahorro del 10 % en kilómetros puede suponer millones de euros al año en flotas grandes.

VARIANTES PRINCIPALES

Del VRP clásico a las variantes reales

  • CVRP

    Capacitated VRP

    Cada vehículo tiene una capacidad máxima de carga (peso o volumen). La variante más común en distribución. La demanda total de cada ruta no puede exceder la capacidad del vehículo asignado.

  • VRPTW

    VRP with Time Windows

    Cada cliente debe ser visitado dentro de una ventana horaria fija (ej.: 9:00–12:00). Añade restricciones temporales que complican enormemente la búsqueda del óptimo global.

  • MDVRP

    Multi-Depot VRP

    Los vehículos pueden partir de varios depósitos distintos. Esencial en redes con múltiples almacenes. Permite asignar cada cliente al depósito más eficiente según proximidad y disponibilidad.

  • SDVRP

    Split Delivery VRP

    Un mismo cliente puede ser servido por más de un vehículo en visitas parciales. Útil cuando la demanda supera la capacidad de un solo camión o cuando se quiere maximizar la ocupación de flota.

COMPARATIVA DE ENFOQUES

Heurísticas clásicas vs IA moderna

ENFOQUES CLÁSICOS

Algoritmos exactos y heurísticas

  • Clarke-Wright Savings: rápido pero subóptimo en instancias grandes (>200 nodos)
  • Simulated Annealing: requiere afinación manual de parámetros temperatura/enfriamiento
  • Branch & Bound: exacto pero inviable para más de 50 nodos en tiempo razonable
  • Ant Colony Optimization: resultados variables entre ejecuciones distintas
  • No aprende de datos históricos ni se adapta a condiciones en tiempo real
IA Y METAHEURÍSTICAS MODERNAS

Aprendizaje + optimización

  • Reinforcement Learning: el agente aprende rutas óptimas por experiencia acumulada
  • Graph Neural Networks: modelan la topología de la red de distribución directamente
  • Pointer Networks: arquitectura Transformer para secuenciación de paradas
  • Hybrid IA + OR: combina predicciones ML con OR-Tools para la optimización final
  • Incorpora patrones históricos, tráfico en tiempo real y predicciones de demanda

CVRP con Google OR-Tools en Python

# CVRP con Google OR-Tools — esquema de producción
from ortools.constraint_solver import routing_enums_pb2
from ortools.constraint_solver import pywrapcp

# 1. Crear el gestor (nodos × vehículos × depósito)
manager = pywrapcp.RoutingIndexManager(
    len(data['distance_matrix']),
    data['num_vehicles'],
    data['depot']
)
routing = pywrapcp.RoutingModel(manager)

# 2. Función de coste (distancia entre nodos)
def distance_callback(from_idx, to_idx):
    i = manager.IndexToNode(from_idx)
    j = manager.IndexToNode(to_idx)
    return data['distance_matrix'][i][j]

transit_cb = routing.RegisterTransitCallback(distance_callback)
routing.SetArcCostEvaluatorOfAllVehicles(transit_cb)

# 3. Restricción de capacidad por vehículo
def demand_callback(idx):
    return data['demands'][manager.IndexToNode(idx)]

demand_cb = routing.RegisterUnaryTransitCallback(demand_callback)
routing.AddDimensionWithVehicleCapacity(
    demand_cb, 0, data['vehicle_capacities'], True, 'Capacity'
)

# 4. Parámetros de búsqueda con metaheurística GLS
params = pywrapcp.DefaultRoutingSearchParameters()
params.first_solution_strategy = (
    routing_enums_pb2.FirstSolutionStrategy.PATH_CHEAPEST_ARC
)
params.local_search_metaheuristic = (
    routing_enums_pb2.LocalSearchMetaheuristic.GUIDED_LOCAL_SEARCH
)
params.time_limit.FromSeconds(30)

solution = routing.SolveWithParameters(params)

OR-Tools. Librería open-source de Google. Soporta CVRP, VRPTW y MDVRP con pocas líneas de configuración. Escala hasta ~1 000 nodos en producción.

Guided Local Search. Metaheurística por defecto para VRP. Escapa de mínimos locales penalizando arcos ya usados frecuentemente en iteraciones anteriores.

OptaPlanner. Alternativa JVM (Java/Kotlin) con motor de constraint satisfaction. Más integrable con sistemas ERP y TMS basados en Java.

Límite de tiempo. En producción se suele configurar 30–120 s. OR-Tools devuelve la mejor solución encontrada hasta ese momento al expirar el límite.

BIN PACKING 3D

El problema del empaquetado 3D

El bin packing 3D consiste en colocar cajas de distintas dimensiones dentro de contenedores minimizando el espacio desaprovechado y cumpliendo restricciones físicas y operativas.

Es NP-hard en su versión tridimensional. Las soluciones industriales combinan heurísticas geométricas con modelos de IA entrenados sobre miles de cargas históricas.

Las ganancias son directas: un camión con el 92 % de ocupación volumétrica frente al 72 % habitual supone un 20 % menos de viajes y la misma reducción en costes de combustible y emisiones.

  • Bottom-Left-Fill (BLF): coloca cada caja en la posición más baja, izquierda y trasera disponible
  • Extreme Points: genera puntos de inserción candidatos a partir de las esquinas de las cajas ya colocadas
  • IA generativa: secuencia la carga y prevé estabilidad bajo frenadas
RESTRICCIONES Y HERRAMIENTAS

Más allá del volumen: carga inteligente

  • 01

    Peso y distribución axial

    El peso máximo por eje del vehículo es una restricción legal. Los algoritmos calculan el centro de gravedad y garantizan que la distribución sea conforme a la normativa de circulación vigente.

  • 02

    Compatibilidad de mercancías

    Algunos productos no pueden compartir espacio: frescos con secos, peligrosas con alimentos, frágiles bajo pesados. Las incompatibilidades se modelan como grafos de restricción binaria.

  • 03

    Orden de entrega LIFO

    La carga debe organizarse de forma que la primera entrega esté más accesible (Last In, First Out). Esto obliga a conocer el orden de ruta antes de planificar la carga — acoplando el VRP con el bin packing.

  • 04

    Software especializado

    Cargo Planner, 3D Load, Packit3D: SaaS con IA y visualización 3D. SAP EWM: gestión de carga en ERP. Covariant AI: robótica + carga automatizada. Logiview / OpenBoxes: opciones open-source para pymes.

CAPÍTULO 03

Predicción de ETA

Modelos de Machine Learning para estimar tiempos de llegada con precisión real — no distancia dividida por velocidad media.

FACTORES Y MODELOS

¿Qué determina el tiempo real de llegada?

El ETA clásico asume velocidades medias constantes. Un modelo de ML incorpora todas las variables que determinan el tiempo real de tránsito en cada segmento de ruta.

Tráfico dinámico: APIs de HERE, TomTom o Google Maps en tiempo real, complementadas con patrones históricos por hora del día, día de la semana y eventos locales (eventos deportivos, obras).

Condiciones meteorológicas: lluvia, niebla, nieve y temperatura afectan la velocidad efectiva. Un modelo entrenado sobre histórico de despachos cuantifica el impacto por tipo de evento climático.

Historial del conductor: cada conductor tiene patrones propios de velocidad y paradas. El modelo aprende estas variaciones individuales y las incorpora a la predicción.

Tiempo en cliente: la variabilidad en el tiempo de descarga (colas, disponibilidad de muelles) es a menudo el mayor contribuyente a la dispersión del ETA real.

VRP básico con OR-Tools — definir distancias y resolver rutas

from ortools.constraint_solver import routing_enums_pb2, pywrapcp

# 1. Datos del problema: 5 clientes + 1 depósito (índice 0)
data = {
    'distance_matrix': [
        [0,  10, 15, 20, 25, 12],
        [10,  0, 35, 25, 30, 18],
        [15, 35,  0, 30, 20, 22],
        [20, 25, 30,  0, 15, 28],
        [25, 30, 20, 15,  0, 14],
        [12, 18, 22, 28, 14,  0],
    ],
    'num_vehicles': 2, 'depot': 0
}
manager = pywrapcp.RoutingIndexManager(
    len(data['distance_matrix']), data['num_vehicles'], data['depot'])
routing = pywrapcp.RoutingModel(manager)

def dist_cb(i, j):
    return data['distance_matrix'][manager.IndexToNode(i)][manager.IndexToNode(j)]

cb = routing.RegisterTransitCallback(dist_cb)
routing.SetArcCostEvaluatorOfAllVehicles(cb)

params = pywrapcp.DefaultRoutingSearchParameters()
params.first_solution_strategy = (
    routing_enums_pb2.FirstSolutionStrategy.PATH_CHEAPEST_ARC)
sol = routing.SolveWithParameters(params)

# 2. Imprimir rutas óptimas
for v in range(data['num_vehicles']):
    idx, ruta = routing.Start(v), []
    while not routing.IsEnd(idx):
        ruta.append(manager.IndexToNode(idx))
        idx = sol.Value(routing.NextVar(idx))
    print(f"Vehículo {v}: {ruta} → depósito")

Bin packing greedy: optimización de cargas y verificación

from ortools.algorithms.python import knapsack_solver

# Pesos (kg) y volúmenes (dm³) de 8 bultos. Camión: 10.000 kg, 50 m³
pesos   = [1200, 800, 3200, 950, 2100, 400, 1750, 600]
valores = [120,  85,  310, 95,  200,  45,  180,  65]
cap_kg  = 10000

# Greedy simple: ordenar por densidad de valor (valor/peso)
bultos = sorted(zip(valores, pesos, range(len(pesos))),
               key=lambda x: x[0]/x[1], reverse=True)
seleccionados, peso_total, valor_total = [], 0, 0
for v, p, i in bultos:
    if peso_total + p <= cap_kg:
        seleccionados.append(i); peso_total += p; valor_total += v

print(f"Greedy — bultos: {seleccionados}, peso: {peso_total} kg, valor: {valor_total}")

# Verificación exacta con OR-Tools knapsack (óptimo garantizado)
solver = knapsack_solver.KnapsackSolver(
    knapsack_solver.KNAPSACK_DYNAMIC_PROGRAMMING_SOLVER, 'KP')
solver.Init(valores, [pesos], [cap_kg])
opt = solver.Solve()
opt_items = [i for i in range(len(pesos)) if solver.BestSolutionContains(i)]
print(f"OR-Tools óptimo — bultos: {opt_items}, valor total: {opt}")
print(f"Gap greedy vs óptimo: {((opt-valor_total)/opt)*100:.1f}%")

Dos enfoques. El greedy es instantáneo pero no garantiza el óptimo. OR-Tools calcula el óptimo exacto y te muestra el gap.

Qué observar. El gap suele ser < 5 % para instancias reales. Si el greedy da el mismo valor que el óptimo, ¡felicidades, no hay gap!

Extensión práctica. Añade una segunda restricción [volumenes], [50000] al método Init para modelar la capacidad volumétrica del camión además del peso.

import pandas as pd
import numpy as np
from sklearn.ensemble import GradientBoostingRegressor
from sklearn.model_selection import train_test_split
from sklearn.metrics import mean_absolute_error

# Dataset sintético de 500 viajes — copia y ajusta con tus datos reales
np.random.seed(42)
n = 500
df = pd.DataFrame({
    'distancia_km':  np.random.uniform(10, 400, n),
    'hora_salida':   np.random.randint(0, 24, n),
    'lluvia':        np.random.choice([0, 1], n, p=[0.8, 0.2]),
    'tipo_ruta':     np.random.choice([0, 1, 2], n),  # 0=urbana 1=nacional 2=autopista
    'trafico_idx':   np.random.uniform(0.5, 2.0, n),
})
velocidad_base = [35, 70, 110]
df['eta_min'] = (df['distancia_km'] / df['tipo_ruta'].map({0:35,1:70,2:110})
               ) * 60 * df['trafico_idx'] + df['lluvia']*12

features = ['distancia_km', 'hora_salida', 'lluvia', 'tipo_ruta', 'trafico_idx']
X, y = df[features], df['eta_min']
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

model = GradientBoostingRegressor(n_estimators=100, max_depth=4)
model.fit(X_train, y_train)
preds = model.predict(X_test)
print(f"MAE: {mean_absolute_error(y_test, preds):.1f} minutos")
print(f"Ejemplo predicción: {model.predict([[150, 8, 0, 1, 1.2]])[0]:.0f} min")

Simulación de escenarios de coste con pandas + itertools

import pandas as pd
import itertools

# Rutas activas de la semana (km, horas conductor, tipo vehículo)
rutas = pd.DataFrame({
    'ruta':       ['Madrid-Valencia', 'BCN-Zaragoza', 'Sevilla-Málaga', 'Bilbao-Vitoria'],
    'km':         [356, 307, 212, 68],
    'horas_cond': [4.2, 3.8, 2.6, 1.1],
    'peaje':      [42,  28,  15,  6],
    'consumo_l100': [30, 28, 26, 24],
})

# Escenarios: precio combustible (€/L) × coste conductor (€/h)
combustibles = [1.45, 1.65, 1.85]
coste_conductor = [18, 22, 26]

resultados = []
for precio_fuel, coste_h in itertools.product(combustibles, coste_conductor):
    df = rutas.copy()
    df['coste_fuel'] = df['km'] * df['consumo_l100'] / 100 * precio_fuel
    df['coste_cond'] = df['horas_cond'] * coste_h
    df['coste_total'] = df['coste_fuel'] + df['coste_cond'] + df['peaje']
    resultados.append({'fuel': precio_fuel, 'cond': coste_h,
                       'total_semana': df['coste_total'].sum()})

tabla = pd.DataFrame(resultados).pivot(index='fuel', columns='cond', values='total_semana')
print(tabla.round(0))

import anthropic, json

# Tabla de rutas de la semana (en producción: lee de tu TMS/ERP)
tabla_rutas = [
    {"ruta": "Madrid-Valencia", "km": 356, "coste_total": 318, "ocupacion_pct": 68},
    {"ruta": "BCN-Zaragoza",    "km": 307, "coste_total": 271, "ocupacion_pct": 91},
    {"ruta": "Sevilla-Málaga",  "km": 212, "coste_total": 198, "ocupacion_pct": 55},
    {"ruta": "Bilbao-Vitoria",  "km": 68,  "coste_total": 84,  "ocupacion_pct": 42},
]
tabla_str = json.dumps(tabla_rutas, ensure_ascii=False, indent=2)

client = anthropic.Anthropic()  # ANTHROPIC_API_KEY en variable de entorno
mensaje = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": (
            f"Analiza esta tabla de rutas logísticas y proporciona:\n"
            f"1. Las 2 rutas con mayor oportunidad de ahorro y por qué.\n"
            f"2. Una recomendación concreta para cada una (consolidación, horario, carga).\n"
            f"3. Ahorro estimado en porcentaje si se aplican las medidas.\n\n"
            f"Datos:\n{tabla_str}"
        )
    }]
)
print(mensaje.content[0].text)

Instala con: pip install anthropic

API key. Crea una en console.anthropic.com y ponla en la variable de entorno ANTHROPIC_API_KEY. No la escribas en el código.

Qué hace el LLM. Identifica que Bilbao-Vitoria (42 % ocupación) y Sevilla-Málaga (55 %) son las candidatas a consolidación o cambio de frecuencia, y cuantifica el ahorro potencial.

En producción. Automatiza esta llamada cada lunes por la mañana y envía el análisis por email al jefe de tráfico antes de la reunión de planificación semanal.
