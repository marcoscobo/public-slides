FUNDAMENTOS

Qué es un MVP

Minimum Viable Product: la versión más pequeña de tu producto que entrega valor real a un usuario real.

El MVP no es un producto incompleto. Es un producto completo para un problema muy específico. La diferencia es enorme.

EL RIESGO REAL

El error más común

Construir demasiado.

El mayor riesgo cuando tienes una herramienta tan poderosa es caer en el feature creep. La IA puede construir todo lo que se le pida. Tú debes decidir qué NO pedir.

El MVP se define más por lo que excluye que por lo que incluye.

CAPÍTULO 01

Definir el producto

De la idea vaga al requisito concreto que la IA puede ejecutar.

SCOPE

Cómo definir el scope

Para cada feature potencial, hazte una sola pregunta.

Por ejemplo, para un MVP de gestor de tareas de equipo:

DENTRO DEL MVP

  • Crear tarea
  • Marcar completada
  • Ver lista

BACKLOG

  • Notificaciones
  • Prioridades
  • Categorías / Filtros
  • Integración calendario

Puedes construir todo el backlog después. Con IA, cada feature es cuestión de horas, no semanas.

CLARIDAD DE IDEA

Tres preguntas para clarificar tu idea

  1. 01

    ¿Qué problema específico resuelve?

    No: quiero una red social. Sí: quiero una app para que mi equipo de 5 personas vea qué tarea está haciendo cada uno sin tener que preguntar.

  2. 02

    ¿Para quién exactamente?

    Cuanto más específico, mejor. La especificidad es una ventaja, no una limitación.

  3. 03

    ¿Cuál es el momento mínimo de valor?

    El primer instante en que el usuario experimenta el beneficio. Todo lo demás puede esperar.

REQUISITOS

Requisitos en lenguaje natural

Los requisitos son la traducción de la idea en instrucciones concretas que la IA puede ejecutar.

FORMATO Como [tipo de usuario],
quiero [funcionalidad],
para [beneficio].
EJEMPLO 01

Como miembro del equipo, quiero poder crear una tarea nueva indicando nombre y responsable, para que todo el mundo sepa quién está haciendo qué.

EJEMPLO 02

Como gestor, quiero ver en un panel todas las tareas activas con su estado, para no tener que preguntar en qué está cada persona.

Los user stories son el lenguaje que conecta tu visión con las instrucciones de la IA.

CAPÍTULO 02

El ciclo iterativo

La unidad de trabajo con IA generativa: del requisito al commit en 15–45 minutos.

EL PROCESO

Cinco pasos que se repiten

  • 01

    Definir el requisito

    Escribir la user story en lenguaje natural antes de abrir Claude Code.

  • 02

    Contextualizar el prompt

    Aplicar la estructura de las 5 partes aprendida en el Módulo 04: rol, contexto, restricciones, formato, tarea.

  • 03

    Revisar el plan

    Pedir a la IA que explique qué va a hacer antes de ejecutar. Nunca ejecutar a ciegas.

  • 04

    Ejecutar e inspeccionar

    Probar manualmente como lo haría un usuario real. No confiar sin verificar.

  • 05

    Comprometer o iterar

    Si funciona: git commit. Si no: describir el problema con precisión y repetir desde el paso 01.

DIAGRAMA DEL CICLO

El bucle en la práctica

Este ciclo puede completarse en 15–45 minutos por feature.

Un día productivo son 6–10 ciclos completos.

VELOCIDAD DE ITERACIÓN
15–45

min es lo que tarda un ciclo completo: del requisito al commit probado.

En un día productivo puedes completar 6–10 ciclos: eso es 6–10 features validadas. Sin IA, una sola feature involucraba a varias personas y podía llevar días.

CAPÍTULO 03

Validar sin saber código

Cómo saber si lo que la IA construyó es correcto, seguro y sólido.

VALIDACIÓN

Cinco estrategias de validación

  • 01

    Prueba funcional

    ¿Hace exactamente lo que pediste? Pruébalo como lo haría un usuario real, sin atajos.

  • 02

    Pregunta a la IA

    Explícame en términos simples qué hace este código y qué podría fallar.

  • 03

    Verifica restricciones

    ¿Cumplió todas las restricciones que pusiste? ¿Modificó algo que no debía?

  • 04

    Regresión

    ¿Todo lo que funcionaba antes sigue funcionando tras el último cambio?

  • 05

    Revisión de deuda

    Si un desarrollador profesional viera este código, ¿qué criticaría? Pregúntaselo a la IA.

ARQUITECTURA

La arquitectura mínima

No necesitas saber implementar cada capa. Necesitas saber describírsela a la IA.

TESTING

Testing sin programar

Los tests son el seguro de vida de tu producto. La IA los escribe por ti, pero necesitas entender qué cubren.

TESTS UNITARIOS Escribe tests para la función que calcula el total del carrito.
TESTS DE INTEGRACIÓN Escribe tests que comprueben que el registro funciona de principio a fin.
TESTS MANUALES GUIADOS Dame una lista de casos que debería probar manualmente para validar el login.
HÁBITO CLAVE

En el prompt de cada nueva feature, incluye siempre: y escribe los tests correspondientes.

GESTIÓN DE ERRORES

Cuando algo no funciona

CUANDO LA IA SE BLOQUEA

Qué hacer tú

  • Copia el error completo y pégaselo: "Este es el error que veo: [error]. ¿Qué está pasando y cómo lo resuelvo?"
  • Pide opciones: "Dame 3 formas distintas de solucionar este problema."
  • Simplifica: "Hagamos esto en pasos más pequeños. Primero solo [parte A]."
  • Busca el error exacto en internet (StackOverflow, GitHub Issues).
CUÁNDO PEDIR AYUDA HUMANA

Los límites reales

  • Llevas más de 1 hora atascado en el mismo error.
  • La IA propone soluciones que contradicen el CLAUDE.md del proyecto.
  • El problema parece de infraestructura o de la plataforma, no de tu código.

IMPACTO REAL

Con IA vs sin IA

SIN IA GENERATIVA

Desarrollo tradicional

  • MVP básico: 2–4 semanas.
  • Dependencia total de equipo técnico.
  • Coste externalizado: 5.000–20.000€.
  • Iteraciones lentas: días por feature.
CON IA GENERATIVA

Tu nuevo superpoder

  • MVP básico: 2–5 días.
  • Autonomía total. Tú decides cuándo y qué.
  • Coste de herramientas: ~100€/mes
  • Iteraciones rápidas: horas por feature.
EN NÚMEROS

Lo que cambia

2–5 días para construir un MVP básico funcional desde cero
~100 €/mes coste de todas las herramientas del curso
15–45 min por ciclo iterativo completo: del requisito al commit
0x velocidad de iteración con IA vs desarrollo tradicional
