Control de versiones

POR QUÉ GIT

Git es tu máquina del tiempo

Git guarda una fotografía del proyecto en cada momento importante. Si algo falla, vuelves a cualquier punto anterior en segundos.

Con la IA esto es todavía más valioso: el agente hace cambios grandes sin preguntar. Un commit previo te permite deshacer todo con un solo comando.

CÓMO FUNCIONA

De tu ordenador a GitHub

Git tiene cuatro zonas. Cada comando mueve los cambios de una zona a la siguiente. github.com — crea tu cuenta si aún no tienes.

Working Directory
Donde editas los archivos. Los cambios aún no están registrados por Git.
Staging Area
La sala de espera. Preparas qué va en el próximo commit con git add.
Repositorio Local
El historial en tu máquina. Commits guardados con git commit.
GitHub (Remoto)
La nube. Copia de seguridad y punto de despliegue. Se sube con git push.
LOS COMANDOS ESENCIALES

4 comandos que necesitas dominar

  • 01

    git init / git clone

    Crear un repo nuevo en local o clonar uno existente de GitHub. Es siempre tu punto de partida en cualquier proyecto.

  • 02

    git add + git commit

    Fotografía del estado actual. Hazlo ANTES de pedirle a la IA un cambio grande. Es tu red de seguridad: si algo falla, vuelves aquí.

  • 03

    git push

    Subir los commits a GitHub. Copia de seguridad en la nube y, si tienes despliegue continuo configurado, disparador automático de una nueva versión.

  • 04

    git checkout / git reset

    Volver a una versión anterior. El salvavidas cuando algo se rompe después de una sesión con la IA. La diferencia entre perder horas y perder segundos.

EN LA PRÁCTICA

Tu primer commit

# 1. Crear la carpeta del proyecto y convertirla en repositorio Git
git init my-first-repo
cd my-first-repo

# 2. Añadir todos los archivos a la zona de staging
git add .

# 3. Crear el primer commit con un mensaje descriptivo
git commit -m "primer commit del proyecto"

# 4. Conectar con el repositorio remoto en GitHub
git remote add origin https://github.com/tu-usuario/my-first-repo.git

# 5. Subir el primer commit a la rama principal
git push -u origin main

git init Crea una carpeta oculta .git/ que contiene todo el historial de versiones. Sin esta carpeta, el directorio no es un repositorio.

git add . El punto selecciona todos los archivos modificados. Puedes ser más selectivo: git add src/app.js solo añade ese archivo.

git commit -m Guarda la fotografía con un mensaje que explica qué hay en ese estado. Cuanto más descriptivo, más útil será meses después.

git push -u El flag -u vincula la rama local con la remota. Solo se necesita la primera vez; después basta con git push.

ARCHIVOS ESENCIALES

El archivo que protege tus secretos

.gitignore es una lista de archivos y carpetas que Git debe ignorar completamente. Nunca los incluirá en ningún commit.

Sin este archivo, podrías subir accidentalmente a GitHub tus claves de API, tokens de acceso, contraseñas de bases de datos o carpetas de cientos de megabytes de dependencias.

Se crea en la raíz del proyecto y Git lo lee automáticamente. Lo primero que hace la IA cuando inicializa un proyecto es generar este archivo.

Regla de oro: si no quieres que el mundo lo vea, está en .gitignore. Si necesita viajar con el proyecto, está en el repo.

RAMAS · BRANCHES

Experimenta sin romper nada

La rama main es la versión estable de tu proyecto. Las ramas de feature permiten probar cambios sin afectar a main.

Con IA, usa ramas para experimentar: si Claude Code o Antigravity sugiere una refactorización grande, hazlo en una rama nueva. Si funciona, la fusionas. Si no, la borras sin consecuencias.

git checkout -b feature/mi-experimento

Este es el flujo básico. En el siguiente slide vemos cómo escalar esto al estándar completo de equipos.

GITFLOW · ESTÁNDAR DE EQUIPOS

El flujo completo para proyectos reales

Gitflow es el estándar que usan la mayoría de equipos profesionales. Añade dos ramas permanentes y reglas claras de cuándo hacer cada cosa.

  • main — código en producción. Solo recibe de release o hotfix.
  • develop — integración continua. Features van aquí.
  • feature/* — nuevas funcionalidades desde develop.
  • release/* — preparar una versión. Va de develop a main.
  • hotfix/* — parches urgentes desde main, vuelven a develop.
COMPARATIVA

Sin Git vs con Git

SIN GIT

Trabajar sin red

  • Cada cambio es permanente e irrecuperable
  • La IA rompe algo y no sabes qué era antes
  • Copias manuales: proyecto-v2-FINAL-FINAL.zip
  • Sin historial de por qué se tomó cada decisión
  • Imposible colaborar sin pisar el trabajo del otro
CON GIT

Trabajar con confianza

  • Cada commit es un punto de restauración
  • La IA hace cambios grandes: un comando para deshacer
  • Historial limpio con mensajes descriptivos
  • El porqué de cada decisión queda registrado
  • Colaboración asíncrona sin conflictos

MENSAJES DE COMMIT

Conventional Commits

Los Conventional Commits son un estándar para escribir mensajes de commit con estructura predecible y significado semántico.

El formato es: tipo: descripción. El tipo declara qué clase de cambio es. La descripción explica el qué.

La IA puede generar y mantener este estilo automáticamente. Basta con indicarlo en el CLAUDE.md: "usa Conventional Commits en todos los mensajes".

Tipos de commit
feat: nueva funcionalidad — sube la versión minor
fix: corrección de un bug — sube la versión patch
docs: solo documentación, sin cambios de código
refactor: reorganización sin cambiar comportamiento
test: añadir o corregir tests
chore: tareas de mantenimiento sin impacto
Ejemplos reales:
feat: añadir login con Google
fix: corregir error en el carrito al vaciar
docs: actualizar README con los requisitos

Gratis para proyectos personales y también tu portfolio técnico visible para cualquier reclutador o colaborador.

Organización de proyectos

Un proyecto bien estructurado es como una cocina bien organizada: la IA encuentra los ingredientes donde espera encontrarlos.

ESTRUCTURA ESTÁNDAR

La anatomía de un proyecto bien organizado

Cada archivo tiene una razón de ser. La IA espera encontrar cada pieza en su lugar para trabajar con mayor eficiencia.

El archivo más importante para la IA es CLAUDE.md (o su equivalente en Antigravity): le dice quién eres, qué hace el proyecto, qué convenciones seguir y qué nunca debe hacer.

Antigravity usa un archivo equivalente (el nombre varía por agente, pero el concepto es el mismo). Cada agente tiene su forma de recibir contexto, pero todos lo necesitan.

La carpeta .claude/ contiene agentes especializados en agents/ y habilidades reutilizables en skills/. El equivalente en Antigravity sigue el mismo principio.

CONVENCIONES

Las reglas que hacen que todo encaje

  • 01

    Nombres en inglés para archivos y carpetas

    Los identificadores técnicos en inglés con guiones: user-auth, api-client. La documentación en el idioma que prefieras. La IA trabaja mejor con nombres en inglés.

  • 02

    README.md siempre en la raíz

    Primera cosa que ve cualquier persona (o IA) que entra al proyecto. Sin README, el proyecto no existe para el mundo.

  • 03

    .gitignore desde el primer commit

    node_modules/, .env, __pycache__/, *.log nunca se suben a GitHub. Configúralo antes de hacer el primer commit.

  • 04

    CHANGELOG.md para decisiones importantes

    Registra qué cambiaste y por qué. La IA puede mantenerlo automáticamente con el prompt adecuado en el CLAUDE.md.

CAPÍTULO 04

Documentación que funciona

No documentas para el pasado. Documentas para la IA que trabajará contigo mañana.

EL README

Anatomía de un buen README

El README es el contrato entre tu proyecto y el mundo. En 60 segundos, cualquier persona (o IA) debe entender qué hace y cómo arrancarlo.

Nombre y descripción — Una línea que define el proyecto. Si no puedes resumirlo en una línea, es que el proyecto hace demasiado.

Instalación — El camino más corto posible para que alguien lo tenga corriendo. Idealmente un solo comando.

Uso — Un ejemplo real. No lo que podría hacer el proyecto; lo que hace ahora mismo con un ejemplo concreto.

Tecnologías — El stack: qué lenguajes, frameworks y servicios usa. La IA necesita esta información para generar código compatible.

Contribuir — Las reglas del juego si alguien quiere colaborar o si tú quieres decirle a la IA cómo debe hacer sus PRs.

ARCHIVO ESENCIAL

Lo que nunca debe llegar a GitHub

# === SECRETOS — NUNCA subir claves de API ===
.env
.env.local
.env.production

# === DEPENDENCIAS (todos los lenguajes) ===
node_modules/       # Node.js
__pycache__/        # Python bytecode
*.pyc               # Python compilado
vendor/             # PHP Composer
.gradle/  build/    # Java/Kotlin

# === BUILD (regenerados en cada deploy) ===
dist/   .next/   .nuxt/   out/

# === SISTEMA OPERATIVO ===
.DS_Store   Thumbs.db

# === LOGS Y TEMPORALES ===
*.log   npm-debug.log*

La regla es universal. Cada lenguaje genera carpetas de dependencias que pesan cientos de MB y se pueden regenerar. Nunca se versionan.

.env Si subes este archivo a un repo público, tus claves quedan expuestas. GitHub escanea en busca de claves filtradas y notifica a los proveedores. La llave se revoca automáticamente.

.env.example Este sí se versiona: lista qué variables necesita el proyecto, sin los valores reales. Es la guía para el equipo.

gitignore.io Genera el .gitignore según tu stack en segundos: escribe "node express docker" y lo tienes.

FLUJO DIARIO CON GIT + IA

El ritual que repetirás cada día

  • 01

    git pull — sincronizar antes de empezar

    Traer los cambios del remoto. Si trabajas solo, igualmente es buena costumbre para evitar conflictos.

  • 02

    git checkout -b feature/nombre — nueva rama

    Nunca trabajes directamente en main. La rama protege el código estable de los experimentos del día.

  • 03

    Pedirle al agente la feature — Claude o Antigravity

    Con README y CLAUDE.md bien documentados, el agente genera código de mayor calidad porque tiene contexto real del proyecto.

  • 04

    Probar manualmente — la validación siempre es tuya

    La IA no tiene tu contexto de negocio completo. Revisa que el resultado es lo que querías antes de hacer commit.

  • 05

    git add + commit + push

    Fotografía del estado bueno, mensaje Conventional Commit descriptivo y subida al remoto.

Todo este flujo se lo puedes pedir al agente: crear la rama, hacer el commit, abrir el PR. Es recomendable hacerlo a mano al principio para dominar el control del repositorio. Después decides cuánto delegas.
