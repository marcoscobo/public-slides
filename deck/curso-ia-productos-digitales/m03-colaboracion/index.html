CAPÍTULO 01

Control de versiones

El seguro de vida de cualquier proyecto digital. Y con IA, más que nunca.

POR QUÉ GIT

Git es tu máquina del tiempo

Git es como el historial de versiones de Word, pero para proyectos enteros. Guarda una fotografía del proyecto en cada momento importante.

Con la IA, Git es todavía más valioso: cuando el agente hace cambios grandes, un commit previo te permite deshacer en segundos.

No se trata de ser programador. Se trata de trabajar con seguridad.

CÓMO FUNCIONA

De tu ordenador a GitHub

Git tiene cuatro zonas. Entender cómo fluye el código entre ellas lo aclara todo.

Cada comando mueve los cambios de una zona a la siguiente. La flecha de retorno (git pull) trae los cambios del remoto de vuelta a tu máquina.

LOS COMANDOS ESENCIALES

4 comandos que necesitas dominar

  • 01

    git init / git clone

    Crear un repo nuevo o clonar uno existente. Tu punto de partida en cualquier proyecto.

  • 02

    git add + git commit

    Fotografía del estado actual. Hazlo ANTES de pedirle a la IA un cambio grande. Es tu red de seguridad.

  • 03

    git push

    Subir cambios a GitHub. Copia de seguridad en la nube y punto de despliegue automático.

  • 04

    git checkout / git reset

    Volver a una versión anterior. El salvavidas cuando algo se rompe después de una sesión con la IA.

EN LA PRÁCTICA

Tu primer commit

# 1. Crear la carpeta del proyecto y entrar
git init mi-proyecto
cd mi-proyecto

# 2. Fotografiar el estado actual (todos los archivos)
git add .

# 3. Guardar la fotografía con un mensaje descriptivo
git commit -m "feat: estructura inicial del proyecto"

# 4. Conectar con el repositorio remoto en GitHub
git remote add origin https://github.com/usuario/mi-proyecto.git

# 5. Subir el primer commit a la rama principal
git push -u origin main

Convención de mensajes. El prefijo feat: sigue Conventional Commits. Otros: fix:, docs:, refactor:. La IA puede mantener este estilo por ti.

git add . El punto significa "todos los archivos modificados". Puedes ser más selectivo: git add src/archivo.js

-u origin main. Solo necesitas este flag la primera vez. Después, git push es suficiente.

RAMAS · BRANCHES

Experimenta sin romper nada

La rama main es la versión estable de tu proyecto. Las ramas de feature permiten probar cambios sin afectar a main.

Con IA, usa ramas para experimentar: si Claude Code sugiere una refactorización grande, hazlo en una rama nueva. Si funciona, la fusionas. Si no, la borras sin consecuencias.

git checkout -b feature/mi-experimento — este comando es tu escudo.

COMPARATIVA

Sin Git vs con Git

SIN GIT

Trabajar sin red

  • Cada cambio es permanente e irrecuperable
  • La IA rompe algo y no sabes qué era antes
  • Copias manuales: proyecto-v2-FINAL-FINAL.zip
  • Sin historial de por qué se tomó cada decisión
  • Imposible colaborar sin pisar el trabajo del otro
CON GIT

Trabajar con confianza

  • Cada commit es un punto de restauración
  • La IA hace cambios grandes: un comando para deshacer
  • Historial limpio con mensajes descriptivos
  • El porqué de cada decisión queda registrado
  • Colaboración asíncrona sin conflictos
CAPÍTULO 02

GitHub y el despliegue

Cómo tu código local llega a una URL pública en minutos.

EL ECOSISTEMA

De tu ordenador a una URL pública

GitHub no es solo donde guardas el código. Es donde las plataformas de despliegue buscan tu código para publicarlo automáticamente.

Cada git push dispara un redespliegue automático. Tu URL se actualiza en segundos.

Gratis para proyectos personales. Es también tu portfolio técnico visible para cualquier reclutador o colaborador.

PASO A PASO

Crear tu primer repositorio

  1. 01

    New Repository

    github.com → botón verde "New". Dale un nombre descriptivo en kebab-case.

  2. 02

    Configurar

    Nombre kebab-case, descripción clara, activar README, elegir .gitignore según tecnología.

  3. 03

    git clone

    Copia la URL. En terminal: git clone https://github.com/tu-usuario/mi-proyecto.git

  4. 04

    Abrir en VS Code

    cd mi-proyecto && code . — abre el repo local conectado al remoto en VS Code.

  5. 05

    Claude Code listo

    Abre Claude Code en la carpeta. Pídele estructura, README y .gitignore.

CAPÍTULO 03

Organización de proyectos

Un proyecto bien estructurado es como una cocina bien organizada: la IA encuentra los ingredientes donde espera encontrarlos.

ESTRUCTURA ESTÁNDAR

La anatomía de un proyecto bien organizado

Cada archivo tiene una razón de ser. La IA espera encontrar cada pieza en su lugar.

Un proyecto bien estructurado no solo es más legible para humanos. Es más eficiente para la IA: sabe dónde buscar el contexto, las reglas y el código.

CONVENCIONES

Las reglas que hacen que todo encaje

  • 01

    kebab-case para archivos y carpetas

    mi-proyecto, user-auth, api-client. No MiProyecto ni mi_proyecto. Consistencia absoluta.

  • 02

    Inglés para nombres técnicos

    Los archivos y carpetas en inglés. La documentación en el idioma que prefieras. La IA trabaja mejor en inglés a nivel técnico.

  • 03

    README.md siempre en la raíz

    Primera cosa que ve cualquier persona (o IA) que entra al proyecto. Sin README, el proyecto no existe para el mundo.

  • 04

    .gitignore desde el principio

    node_modules/, .env, __pycache__, *.log nunca se suben a GitHub. Configúralo antes del primer commit.

  • 05

    CHANGELOG.md para decisiones importantes

    Registra qué cambiaste y por qué. La IA puede mantenerlo automáticamente con el prompt adecuado.

CAPÍTULO 04

Documentación que funciona

No documentas para el pasado. Documentas para la IA que trabajará contigo mañana.

EL README

Anatomía de un buen README

El README es el contrato entre tu proyecto y el mundo. En 60 segundos, cualquier persona (o IA) debe entender qué hace y cómo arrancarlo.

La IA puede generar el README inicial a partir de tu descripción del proyecto. Tú lo revisas y enriqueces con detalles que solo tú conoces.

Un README bien escrito reduce las preguntas al equipo a cero y acelera el onboarding de nuevos colaboradores.

ARCHIVO ESENCIAL

Lo que nunca debe llegar a GitHub

# Dependencias (pueden instalarse con npm install)
node_modules/

# Variables de entorno y secretos — NUNCA subir claves
.env
.env.local
.env.production

# Archivos del sistema operativo
.DS_Store
Thumbs.db

# Build y distribución generada
dist/
build/
.next/

# Logs (ruido, no información útil)
*.log
npm-debug.log*

# Instrucciones privadas para la IA (opcional)
.claude/

node_modules/ Puede pesar cientos de MB. npm install lo recrea en segundos. Subirlo es el error más común de principiantes.

.env Contiene tus claves de API, contraseñas y tokens. Subir este archivo a un repo público expone tus credenciales al mundo.

.env.example Sí debes subir este archivo: lista qué variables se necesitan, sin los valores reales. Guía al equipo sin exponer secretos.

DEMO EN VIVO

Repositorio + primer commit con Claude Code

Desde cero hasta proyecto con estructura limpia en GitHub. En 15 minutos.

PASO 1 — Crear repo en GitHub: nombre kebab-case, README activado, .gitignore Node.js.

PASO 2 — Clonar en local: git clone <url> && cd mi-proyecto && code .

PASO 3 — Prompt a Claude Code: "Inicializa este proyecto con estructura estándar, README descriptivo para una app de gestión de tareas, y .gitignore para Node.js".

PASO 4 — Revisar archivos generados: README.md, .gitignore, src/, docs/, CLAUDE.md.

PASO 5 — git add . && git commit -m "feat: estructura inicial" && git push

FLUJO DIARIO CON GIT + IA

El ritual que repetirás cada día

  • 01

    git pull

    Traer los cambios del remoto antes de empezar. Si trabajas solo, igualmente es buena costumbre.

  • 02

    git checkout -b feature/nueva

    Crear una rama para el trabajo del día. Nunca trabajes directamente en main.

  • 03

    Pedirle a Claude Code la feature

    Con README y CLAUDE.md bien documentados, el agente genera código de mayor calidad.

  • 04

    Probar manualmente que funciona

    La IA no tiene tu contexto de negocio completo. La validación final es siempre tuya.

  • 05

    git add + git commit

    Fotografía del estado bueno. Mensaje descriptivo que explique el qué y el por qué.

  • 06

    git push (y PR si en equipo)

    Subir cambios. En equipo, abrir un Pull Request para revisión antes de fusionar a main.
