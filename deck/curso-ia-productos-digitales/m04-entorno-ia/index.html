CAPÍTULO 01

DevContainers

Mismo entorno para todos, siempre. Nueva persona en el equipo, cero horas de configuración.

CONCEPTO

Tu proyecto como entorno reproducible

VS Code lo lee y levanta el entorno en segundos. Cualquiera que clone tu repo tendrá exactamente el mismo entorno que tú.

Un DevContainer es un contenedor Docker que define el entorno de desarrollo completo de tu proyecto: sistema operativo, herramientas, extensiones de editor y comandos de inicialización, todo en un archivo de configuración versionado junto a tu código.

Cuando alguien clona el repositorio y abre VS Code, el editor detecta el archivo .devcontainer/devcontainer.json y ofrece levantar el entorno automáticamente. En dos minutos tienen Node.js, Python, Git, Claude CLI y todas las extensiones instaladas, exactamente igual que tú.

COMPARATIVA

Sin DevContainer vs Con DevContainer

SIN DEVCONTAINER

"En mi máquina funciona"

  • Horas de configuración para cada nuevo colaborador
  • Diferencias entre entornos de dev y producción
  • "Instala la versión 18, no la 20" — mensajes en Slack
  • La IA no sabe qué herramientas tiene disponibles
  • Onboarding doloroso cada vez que entra alguien nuevo
CON DEVCONTAINER

Mismo entorno. Siempre.

  • Nueva persona → clona repo → abre VS Code → listo en 2 min
  • Entorno idéntico al de producción, sin sorpresas
  • Todas las herramientas preinstaladas y versionadas
  • La IA conoce exactamente el entorno y puede aprovecharlo
  • Onboarding que escala sin esfuerzo manual
ANATOMÍA DEL ARCHIVO

Las 5 secciones de un devcontainer.json

  • 01

    name — Identificador del entorno

    "my-project-dev" — Solo para identificación en VS Code. Descriptivo y en inglés.

  • 02

    image — Imagen base del contenedor

    "mcr.microsoft.com/devcontainers/javascript-node:20" — La imagen oficial con Node.js 20 ya instalado.

  • 03

    features — Herramientas adicionales

    git, docker-in-docker, github-cli. Cada feature es una línea: el contenedor las instala automáticamente al crear.

  • 04

    customizations.vscode.extensions — Extensiones del editor

    ["ms-azuretools.vscode-docker", "eamodio.gitlens", "usernamehw.errorlens"] — Se instalan automáticamente al abrir en contenedor.

  • 05

    postCreateCommand — Comando de inicialización

    "npm install && echo 'Entorno listo'" — Se ejecuta una sola vez cuando el contenedor se crea por primera vez.

CÓDIGO REAL · COPIA Y USA

devcontainer.json completo

{
  "name": "hola-mundo-express",
  "image": "mcr.microsoft.com/devcontainers/javascript-node:20",
  "features": {
    "ghcr.io/devcontainers/features/git:1": {},
    "ghcr.io/devcontainers/features/github-cli:1": {}
  },
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-azuretools.vscode-docker",
        "eamodio.gitlens",
        "usernamehw.errorlens"
      ]
    }
  },
  "postCreateCommand": "npm install"
}

image. Imagen oficial de Microsoft con Node.js 20 LTS. Cambia el número para fijar otra versión.

features. Git y GitHub CLI se añaden sobre la imagen base. Cada feature es un módulo versionado independiente.

extensions. Estas tres extensiones se instalan automáticamente para todos los que abran el proyecto en contenedor.

postCreateCommand. npm install se ejecuta una sola vez al crear el contenedor. Añade aquí cualquier setup inicial.

CAPÍTULO 02

CLAUDE.md

El onboarding que le harías a un nuevo colaborador. Sin él, la IA asume. Con él, la IA sabe.

QUÉ ES CLAUDE.md

El archivo que le da identidad a tu proyecto

Cada vez que abres Claude Code en un proyecto, lee CLAUDE.md antes de hacer nada. Es tu oportunidad de definir quién eres, qué estás construyendo y cómo quieres que trabaje.

Lo vemos sobre Claude Code porque es el más ampliamente utilizado en el curso, pero todos los agentes tienen su archivo equivalente. El principio es el mismo: darle contexto persistente antes de trabajar.

LAS 5 SECCIONES CLAVE

Anatomía de un CLAUDE.md efectivo

  • 01

    Descripción del proyecto

    Qué es, para quién y qué problema resuelve. 2-3 párrafos. Es lo primero que lee la IA: debe quedar claro el propósito.

  • 02

    Stack tecnológico

    Todas las herramientas, versiones y frameworks. La IA sabrá exactamente qué puede usar y qué no inventarse.

  • 03

    Convenciones de equipo

    Nomenclatura de archivos, idioma de comentarios, estructura de carpetas preferida. Consistencia automática en cada línea generada.

  • 04

    Reglas de trabajo

    Qué no debe cambiar sin preguntar. Cómo gestionar errores. Cuándo hacer commits. Las restricciones que salvan proyectos.

  • 05

    Contexto de negocio

    Quién es el usuario final, cuál es el objetivo del producto y qué es prioritario. La IA toma mejores decisiones cuando entiende el para qué.

EJEMPLO REAL · PLANTILLA

CLAUDE.md completo

# hola-mundo-express

App web en Node.js + Express que muestra "Hola mundo"
en el puerto 3000, contenedorizada con Docker.

## Stack
- Backend: Node.js 20 + Express 4.x
- Contenedores: Docker + docker-compose.yml
- Deploy: Vercel (configurado en vercel.json)
- Agente IA: Claude Code / Antigravity

## Convenciones
- Archivos en inglés con guiones (my-file.js)
- Variables de entorno NUNCA hardcodeadas
- Commits con Conventional Commits (feat:, fix:…)

## Reglas
- Antes de un cambio grande, explícame el plan primero
- No modificar docker-compose.yml sin confirmación
- Hacer git commit por cada feature completada

Descripción clara. Dos frases que definen qué es. La IA ya sabe el contexto antes de leer nada más.

Versiones explícitas. Express 4.x, Node.js 20 — sin versiones la IA puede usar APIs que no existen en tu entorno.

Reglas que protegen. "No modificar docker-compose.yml sin confirmación" evita cambios destructivos por buena fe.

CAPÍTULO 03

Agentes especializados

Para proyectos complejos, múltiples agentes especializados son más efectivos que uno generalista. Divide y vencerás.

ARQUITECTURA DE AGENTES

Un equipo de IA especializado

Claude Code actúa como orquestador: delega cada tipo de tarea en el agente que mejor la conoce, con instrucciones y criterio propios.

Los agentes se definen como archivos .md en .claude/agents/ con instrucciones específicas para ese rol. Claude Code orquesta entre ellos según el tipo de tarea.

Un agente arquitecto diseña la estructura. Un agente implementador escribe el código. Un agente tester verifica que funciona. Un agente documentador mantiene el README actualizado. Cada uno hace una cosa y la hace bien.

DEFINICIÓN DE AGENTE · .claude/agents/

Anatomía de un agente especializado

---
name: code-reviewer
description: Revisa el código generado buscando bugs,
  problemas de seguridad y mejoras de calidad.
---

Eres un revisor de código experto.
Cuando revises código:

1. Busca bugs potenciales antes de que lleguen a producción
2. Identifica problemas de seguridad (inyección SQL, XSS, etc.)
3. Sugiere mejoras de legibilidad sin cambiar la lógica
4. Verifica que se siguen las convenciones del CLAUDE.md
5. Si algo está bien, dilo explícitamente — no solo señales errores

Frontmatter YAML. name y description son los metadatos que Claude Code usa para decidir cuándo invocar este agente.

Instrucciones de rol. La sección tras el --- define la personalidad y el comportamiento. Cuanto más específico, mejor resultado.

Lista de tareas. Numerar las responsabilidades ayuda al agente a no olvidar ningún paso en revisiones largas.

SKILLS PERSONALIZADOS

Skills: flujos de trabajo en un comando

En lugar de escribir el mismo prompt largo cada vez, defines la skill una vez y la invocas con /nombre.

Las skills son comandos reutilizables que encapsulan un flujo de trabajo completo. Se definen en .claude/skills/ como archivos .md con las instrucciones del proceso.

/deploy

Verifica tests, despliega a Vercel y actualiza el CHANGELOG automáticamente.

/review

Revisa el código pendiente de commit: bugs, seguridad y convenciones del CLAUDE.md.

/test-all

Ejecuta la suite de tests, agrupa los fallos y propone correcciones.

/document

Actualiza README y comentarios para todos los archivos modificados desde el último commit.

CREATE-SKILL · SKILL ESPECIAL

La skill que crea otras skills

Hay una skill especial: la que te ayuda a descubrir qué skills necesitas y a crearlas. La instala Claude en tu proyecto cuando se la pides.

Una vez instalada, puedes pedirle que analice tu flujo de trabajo habitual y proponga qué procesos merece la pena convertir en skills reutilizables.

El proceso es simple: describes el flujo de trabajo una vez, la skill lo convierte en un comando invocable que cualquier miembro del equipo (o tú en otro contexto) puede usar sin tener que recordar los detalles.

Para instalarla: claude "crea una skill llamada create-skill que me ayude a descubrir y crear skills útiles para este proyecto"

Anatomía de un Prompt Efectivo

Un prompt vago produce resultados vagos. Un prompt con estructura produce código preciso.

LOS 5 ELEMENTOS DEL PROMPT

Los 5 bloques de un prompt que funciona

Un prompt vago produce resultados vagos. Un prompt con estructura produce código preciso y predecible.

Cada prompt efectivo tiene cinco componentes. Puedes usarlos todos o solo los que aplican a tu caso, pero conocerlos te da un checklist mental antes de enviar.

EN LA PRÁCTICA

Prompt vago vs Prompt efectivo

PROMPT VAGO

"Añade un login a mi web."

  • La IA hace suposiciones sobre el stack
  • Puede usar la librería equivocada
  • Puede crear una tabla users diferente a la existente
  • Resultado: código que hay que reescribir
  • Tiempo perdido: 30-60 minutos de correcciones
PROMPT EFECTIVO

"Mi web usa Node.js + Express + PostgreSQL. Necesito autenticación por email con bcrypt. No cambies la tabla users. Tras el login debe redirigir a /dashboard. Antes de escribir código, explícame el plan."

  • Contexto: Node.js + Express + PostgreSQL
  • Objetivo: autenticación por email con bcrypt
  • Restricción: no cambiar tabla users existente
  • Criterio de éxito: redirige a /dashboard
  • Formato: plan de archivos antes de código
PAUSA · REFLEXIÓN

¿Qué elemento del prompt olvidas más frecuentemente?

Acabamos de ver los 5 bloques de un prompt efectivo. Identifica cuál es tu talón de Aquiles.

  • AContexto — No siempre explico qué hay ya en el proyecto antes de pedir algo nuevo
  • BRestricciones — Olvido decirle qué no debe tocar o cambiar
  • CCriterio de éxito — No defino cómo sé que el resultado es correcto
  • DFormato del output — No especifico qué archivos crear o qué estructura quiero
